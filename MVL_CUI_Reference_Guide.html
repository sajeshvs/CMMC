CUI Data Flow Architecture

Target State (M365 GCC G5)

Four concentric defense layers protect CUI from identity through endpoint, data, and collaboration — implementing Zero Trust principles throughout.

Controls Enabled

  • MFA enforced for all CUI users (hardware token for admins)
  • Conditional Access policies: device compliance, location, risk-based
  • No legacy authentication protocols
  • Privileged Identity Management (PIM) — time-bound admin access
  • Microsoft Entra ID Protection — risk-based sign-in analysis

💻 Layer 2 — Endpoint Protection (Intune + Defender XDR)

  • Intune device compliance mandatory before CUI access
  • BitLocker full-disk encryption on all endpoints
  • Defender for Endpoint EDR — continuous monitoring
  • USB write restrictions enforced via Intune policy
  • No personal device access to CUI resources
  • Windows Security Baseline via Microsoft SCT
  • Application whitelisting active
  • Windows Hello for Business (passwordless)
  • Automated patch management via Windows Update for Business
  • Remote wipe capability for lost/stolen devices

🛡 Layer 3 — Data Protection (Microsoft Purview)

Sensitivity Labels

  • CUI — Full encryption, watermark, restricted access
  • CUI-Redacted — Redacted content, wider internal sharing
  • Confidential — Business-sensitive, not federal CUI
  • Internal — General internal use
  • Public — No restrictions

DLP Policies Active

  • Block CUI-labeled content to non-approved domains
  • Alert on any external sharing of CUI content
  • Block downloads to unmanaged devices
  • Prevent copy/paste of CUI to non-approved apps
  • Unified Audit Log — all CUI actions recorded

💬 Layer 4 — Collaboration Security (Teams + SharePoint)

  • Dedicated CUI SharePoint Site Collection — logically isolated within GCC tenant
  • Private Teams only — no public channels for CUI discussion
  • Guest access controlled via GCC B2B — conditional access enforced
  • No public/anonymous sharing links permitted
  • Library-level permission separation (Pre-Award, Post-Award, Redacted, Finance)
  • Retention policies aligned with DoD/federal requirements

Data Loss Prevention (DLP) Policies — Complete Guide

DLP policies work alongside sensitivity labels to prevent unauthorized exfiltration of CUI across Exchange Online, SharePoint, OneDrive, Teams, and endpoints.

DLP Policy Name Scope Condition Action Alert
CUI External Email Block Exchange Online Sensitivity label = CUI AND recipient domain ≠ approved list Block send + notify user High — IT Security + Sender
CUI External Share Block SharePoint / OneDrive Sensitivity label = CUI AND sharing outside org Block sharing + override with justification High — IT Security
CUI Unmanaged Device Block SharePoint / OneDrive Sensitivity label = CUI AND device not Intune-compliant Block download, allow view only Medium — IT Security
CUI Teams Chat Protection Teams CUI keywords / SIT patterns in chat/files Block external recipients, warn internal Medium — IT Security
CUI Endpoint Protection Endpoint DLP CUI-labeled files copied to USB/cloud/print Block USB copy, audit cloud upload, restrict print High — IT Security + Manager
CUI SIT Detection Exchange + SharePoint + OneDrive Sensitive Info Types: DFARS clause numbers, contract numbers, CUI markings Auto-apply CUI label suggestion + block if unlabeled external Medium — Compliance
Unlabeled Content Restriction Exchange + SharePoint No sensitivity label applied AND external recipient Block send/share + prompt to label Low — User notification
⚠
CUI Spillage Prevention: When CUI touches locations not provisioned with adequate protections, that constitutes spillage. The DLP policies above form a defensive perimeter that prevents CUI from leaving the GCC boundary or reaching unauthorized users. All DLP violations are logged in Microsoft Sentinel for forensic analysis and incident response.

📋 DLP Policy Configuration Steps for IT

1

Navigate to Microsoft Purview Compliance Portal

Go to compliance.microsoft.com (or compliance.microsoft.us for GCC) → Data Loss Prevention → Policies → Create Policy

2

Select Template or Custom Policy

Choose "Custom policy" for CUI-specific rules. Name it clearly (e.g., "MVL-DLP-CUI-External-Email-Block").

3

Define Locations

Select Exchange email, SharePoint sites (scope to CUI Site Collection), OneDrive accounts (CUI group members), Teams chat/channels, and Devices (Endpoint DLP).

4

Configure Conditions

Add condition: "Content contains sensitivity label = CUI" AND "Content is shared outside organization" or "Recipient domain NOT IN approved list." Add Sensitive Info Types for CUI keywords, DFARS clause numbers, and contract identifiers.

5

Set Actions & Notifications

Action: Block content sharing. Notifications: Policy tip to user explaining why blocked. Incident report to IT Security distribution group. Override: Require business justification (for approved exceptions only).

6

Test in Simulation Mode First

Always run new DLP policies in simulation mode (Test mode) for 2-4 weeks. Review false positives. Fine-tune conditions. Then switch to enforcement mode.

Automatic Sensitivity Labeling — Configuration Guide

Auto-labeling uses two complementary methods — client-side (within Office apps) and service-side (background processing) — to ensure CUI content is always labeled correctly, even when users forget.

💻 Client-Side Auto-Labeling

Configured within the sensitivity label itself. Triggers in real-time as users create or edit documents in Word, Excel, PowerPoint, and Outlook.

How to Configure

  1. Go to Purview → Information Protection → Labels
  2. Edit the CUI label → Auto-labeling for files and emails
  3. Add conditions:
    • Sensitive Info Types: Custom SITs for CUI identifiers
    • Keywords: "CONTROLLED UNCLASSIFIED", "CUI//", "DISTRIBUTION STATEMENT", "DFARS 252.204"
  4. Set action: Recommend label (users see a banner prompting them to apply) or Auto-apply (no user interaction needed)
  5. Add message: "This document appears to contain CUI. The CUI sensitivity label has been recommended."
💡
Best Practice: Start with "Recommend" mode for user awareness, then shift to "Auto-apply" once users are trained and false positives are tuned.

☁ Service-Side Auto-Labeling

Configured as a standalone policy in Purview. Runs asynchronously in the background across SharePoint, OneDrive, and Exchange. Can process existing at-rest content.

How to Configure

  1. Go to Purview → Information Protection → Auto-labeling
  2. Click + Create auto-labeling policy
  3. Name: "MVL-AutoLabel-CUI-Detection"
  4. Select locations: CUI SharePoint Site Collection, CUI users' OneDrive, Exchange Online
  5. Define rules:
    • Content matches: Custom SITs + keyword lists
    • Document properties: metadata containing CUI markers
    • Sender domain = .mil or .gov (for emails)
  6. Label to apply: CUI
  7. Run in Simulation mode first — review matched items
  8. After validation, switch to Enforce mode
✅
Advantage: Service-side can scan PDFs, images (with OCR), and Exchange attachments. No dependency on Office app versions. Ideal for catching CUI in legacy or bulk-imported content.

Manual Sensitivity Labeling — User Procedures

All CUI users must understand how to manually apply, verify, and manage sensitivity labels. Mandatory labeling is enforced — no document or email can be saved/sent without a label.

📄 Labeling in Office Desktop Apps

  1. Open document in Word, Excel, or PowerPoint
  2. Click the Sensitivity button in the ribbon (Home tab)
  3. Select the appropriate label:
    • CUI for any controlled unclassified content
    • CUI-Redacted for sanitized versions
    • Confidential for business-sensitive (non-CUI)
  4. If downgrading a label, you must provide justification
  5. Labels persist when the document is shared or moved

📧 Labeling in Outlook

  1. Compose new email or reply
  2. Click Sensitivity in the message ribbon
  3. Select the label matching the content sensitivity
  4. Attachment inheritance: If you attach a CUI-labeled document, the email automatically inherits the CUI label
  5. DLP policy tip will appear if sending CUI to external recipients not on the approved domain list

🌐 Labeling in SharePoint Online / Teams

  • Uploading: When uploading files to CUI SharePoint libraries, the library's default label applies automatically. Verify the label is correct in the file properties panel.
  • Creating: New documents created in CUI libraries inherit the library's sensitivity label.
  • Teams files: Files shared in CUI Teams channels are stored in the associated SharePoint library and follow the same labeling rules.
  • Bulk labeling: IT admins can use SharePoint Bulk Label tool or PowerShell to apply labels to existing unlabeled content in CUI libraries.

CUI Incident Response Procedures

🚨 CUI Spillage / Unauthorized Disclosure Response

1

Immediate Containment (Within 1 Hour)

Isolate affected accounts/devices. Revoke sharing links. Quarantine affected files using Purview eDiscovery hold. Disable compromised guest accounts.

2

Assessment (Within 4 Hours)

Determine scope of spillage using Unified Audit Log and Sentinel. Identify which CUI was exposed, to whom, and through what channel. Classify severity: Minor (internal mislabel), Major (external exposure), Critical (adversary access).

3

Notification (Within 72 Hours for DFARS)

Notify Contracting Officer per DFARS 252.204-7012. Report to DC3 (DoD Cyber Crime Center) if cyber incident. Notify affected personnel and CUI Program Manager. Document incident in incident tracking system.

4

Remediation & Lessons Learned

Remove CUI from unauthorized locations. Update DLP rules / Conditional Access to prevent recurrence. Conduct user retraining if human error. Update SSP and POA&M as needed. Close incident with full documentation.

Subcontractors must have their own CUI handling procedures documented and verified. Consider requiring a flow-down clause in subcontracts mandating adequate CUI protections (DFARS 252.204-7012).
